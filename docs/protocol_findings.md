# 雷达协议核对结论 v0.1

- 正式依据：`D:\Drone\wrj\雷达对外协议_2025_含目标点迹协议 (2).doc`
- 核对范围：目标航迹数据及其雷达自身状态、目标状态、校验/帧尾
- 结论级别：字段顺序和位定义已从原文确认；网络字节序和若干语义仍待实报文/厂家确认

## 1. UDP 通信参数

协议给出的默认参数：地面站主机 `192.168.0.9:6000`，雷达 `192.168.0.99:8080`。协议同时说明不同客户网段可能不同，应以实际雷达为准，因此这些值只能作为配置默认值，不能写死。

## 2. 目标航迹帧结构

协议把每个字定义为 32 位整数（`Uint`/`int`）。以下索引为协议中的 1 基字序号：

| 字序号 | 内容 | 正式定义 |
|---:|---|---|
| 1 | 帧头 | `0xA3A3AAAA` |
| 2 | 本帧长度 | 从帧头到帧尾的 32 位字数 |
| 3 | 航迹数目 N | `[0, 500]` |
| 4 | DSP 时间 | ms，航迹运算基准时间 |
| 5 | GPS 时刻 | 协议文字需进一步确认其“天内时间”单位表述 |
| 6 | 雷达 GPS 经度 | `int`，分辨率 `0.00001°`，范围 `[-180,+180]` |
| 7 | 雷达 GPS 纬度 | `int`，分辨率 `0.00001°`，范围 `[-90,+90]` |
| 8 | 雷达海拔/卫星数 | Bits31:16 海拔（m），Bits15:0 卫星数 |
| 9 | 雷达航向角 | `[0,36000]`，单位 `0.01°` |
| 10 | 雷达真航向角 | `[0,36000]`，单位 `0.01°` |
| 11 | 雷达横滚角 | `int`，单位 `0.01°` |
| 12 | 雷达俯仰角 | `int`，单位 `0.01°` |
| 13-18 | 其他状态/备用 | 需按原表保留，不作为首版核心业务输入 |
| 19 | 数处帧数 | `frame_cnt`，显控显示用 |
| 20 | 数处帧时间 | `frame_time`，单位 us |
| 21 起 | N 个目标块 | 每目标固定 16 个 32 位字 |
| 倒数第 2 | 校验和 | 见第 4 节 |
| 最后 1 | 帧尾 | `0xAAAA3A3A` |

总字数必须满足：`N * 16 + 22`。

## 3. 单个航迹目标块

令目标块内索引为 0..15；第 i 个目标从整帧 `word[20 + i*16]`（0 基）开始：

| 块索引 | 正式内容 | 解码要点 |
|---:|---|---|
| 0 | 航迹显示编号 | 面向界面显示 |
| 1 | 航迹绝对编号 | 范围 `[1, 2^32-1]`；内部关联主键 |
| 2 | 距离 | 无符号，单位 0.1 m |
| 3 | NED 角度 | 高 16 位：俯仰角×10，有符号；低 16 位：方位角×10 |
| 4-6 | 备用 | 不解释但保留原值 |
| 7 | 航迹信息 | Bits21:19 目标类型；Bits15:0 目标速度，signed short，单位 0.1 m/s |
| 8 | 备用 | 保留 |
| 9 | TAS/TWS 标志 | Bits1:0；不是“实时点”标志 |
| 10 | GPS 时间 | 单位 ms |
| 11 | 航迹状态/原始类型/DSP 时间戳 | Bit31：0 正常显示、1 清除显示；Bits26:24 原始点迹类型；Bits23:0 DSP 时间标志（ms） |
| 12 | 当前航迹 GPS 经度 | `int`，分辨率 `0.00001°` |
| 13 | 当前航迹 GPS 纬度 | `int`，分辨率 `0.00001°` |
| 14 | 备用 | 高低 16 位均备用 |
| 15 | 目标高度 | Bits15:0，`short`，单位 m；Bits31:16 备用 |

目标类型枚举已确认至少包括：0 未知、1 车辆、2 行人、3 无人机，4 及以上由协议保留/扩展。程序不得据此自动选择类型 3。

## 4. 校验和

协议原文规则：帧头到帧尾的所有 `Uint` 相加，包含帧头和帧尾；计算时先把校验和字段置 0；忽略 32 位溢出；所得 32 位结果写入倒数第 2 个字。

实现应验证：

1. 数据长度是 4 的倍数。
2. 帧头正确。
3. 长度字段等于实际字数。
4. `length == N*16+22`。
5. 帧尾正确。
6. 重算校验和等于报文校验字段。
7. 每个经纬度在物理范围内。

## 5. 已确认的旧源码正确点

- `TRACK_HEADER`、`TRACK_TAIL` 常量与协议一致。
- 1 个目标使用 38 字，符合 `1*16+22`。
- 目标块 16 字，起始位置为固定头 20 字之后。
- 显示编号、绝对编号、距离、目标类型、速度、经纬度和高度的主要块索引与协议一致。
- 模拟器校验和算法包含帧尾并跳过校验字段，和协议一致。

## 6. 旧源码与正式协议/新需求的偏差

| 问题 | 旧行为 | 影响/要求 |
|---|---|---|
| 字节序 | 对同一数据先试小端再试大端 | 生产解析不得靠“哪个能出目标”决定；需配置并确认 |
| 完整性校验 | 解析路径未强制验证帧尾和校验和 | 可能接受截断、损坏或拼接错误的帧 |
| 消除状态 | 未读取目标块索引 11 的 Bit31 | 无法正确触发目标丢失 |
| 状态误读 | 把块索引 9 的 Bits1:0 当作 `is_real_point` | 该字段实际是 TAS/TWS 标志 |
| 航迹关联 | 使用显示编号作为 worker 字典键 | 显示编号可能复用；应以绝对编号为主键 |
| 目标过滤 | 只保留类型 3 | 与人工判断、多类型显示要求冲突 |
| 方位角 | 把雷达航向角再次加到 NED 方位角 | 协议称目标角度为 NED 坐标系信息，旧换算缺少协议依据，暂不得沿用 |
| 雷达海拔 | 对高 16 位直接符号扩展 | 协议行数据类型为 `Uint`，未明确海拔子字段有无符号，应确认 |
| 高度语义 | 直接当相对高度并与 Home 海拔相加 | 正式雷达协议只写“目标高度信息，m”；基准未说明 |

## 7. 不能擅自确定的事项

- UDP 线上 32 位字节序。
- 一个 UDP 数据报是否始终只含一帧、可能含多帧，或可能出现跨数据报分片。
- 雷达海拔高 16 位的有无符号解释。
- 目标高度的基准：相对地面、海拔、相对雷达或其他。
- GPS 时刻字段的精确定义和时区/日滚转规则。
- Bit31“清除显示”是否只发送一次，以及目标未出现多久可判为丢失。
- 航迹绝对编号是否会在雷达重启后复用。
- 雷达坐标系是否明确为 WGS84；协议仅称 GPS 经纬度，未写坐标基准名称。

这些事项必须通过厂家书面答复、真实报文抓包和实物联调确认。
